「出力を見てフィードバックしているのに、なぜか同じ問題が繰り返される」——そういう体験、ありませんか？

AIキャラクターの調整でも、コード生成でも、これはよくあることです。この感覚に名前をつけた論文を読みました。

論文：**The Observability Gap: Why Output-Level Human Feedback Fails for LLM Coding Agents**（arXiv:2603.26942, CHI 2026 Workshop on Human-Agent Collaboration）

---

# 論文の要約

## 何を調べたか

LLMマルチエージェントコーディングシステムでは、通常エージェントの能力は設計時に固定されます。この論文は代わりに**「獲得された自律性（earned autonomy）」**という設定を研究しました。

ルールは：
- エージェントは最初ゼロの事前定義済み関数から始まる
- **人間はビジュアル出力だけを見てフィードバックを与える**
- フィードバックをもとにエージェントが再利用可能な関数ライブラリを自分で積み上げていく

タスクは**Blenderを使った3Dシーン生成**。空間的な推論とプログラム的な幾何学制御が両方必要な、かなり複雑な課題です。

## 何が起きたか

結果は衝撃的でした：**複数の指示粒度で試しても、出力のみのフィードバックでは完全シーン成功率0%**。

「完全な成功」の定義は：
- オブジェクトの完全性（必要なものが全部ある）
- 地面との接触（物体がちゃんと床に乗っている）
- 衝突回避（物体が重ならない）
- スケールの妥当性（大きさがおかしくない）

これを全部同時に満たす必要があります。エージェントは人間の実装と同等のユーティリティ関数を**自力で再発見**したにもかかわらず、完全成功はゼロ。

## なぜ直らないのか：観測可能性ギャップ

論文が提示する説明がこれです：

```
バグの発生源：コードロジック・実行状態（内部）
人間の評価場所：ビジュアル出力（外部）
```

この二層の間に**「観測可能性ギャップ（observability gap）」**があります。

問題の核心は「多対一写像」です。**内部状態の組み合わせは無数にあるが、見える出力は一つ**。同じように見える失敗が、全く異なる原因から来ている可能性があります。

これが何を起こすか：
- 「床に物体が接していない」という症状を見る
- ← でも原因は座標計算のバグかもしれないし、スケール設定のバグかもしれないし、関数の呼び出し順序のバグかもしれない
- 症状レベルのフィードバックでは根本原因を特定できない
- エージェントは対処を繰り返すが、収束せず**失敗モードが振動する**

## 介入実験で確かめた

重要なのが「診断的介入」の実験です。

最小限のコードレベルの知識を注入したところ——**収束が回復した**。

これが示すこと：問題はエージェントのプログラミング能力ではなく、**フィードバックの観測可能性がボトルネック**だということです。

## 論文の結論：フィードバックパラドックス

論文はこれを「フィードバックパラドックス」として定式化します：

> コードロジックと知覚的結果の間に深い因果連鎖がある領域では、出力のみの評価では効果的な人間-エージェント協調ができない。

解決策として論文が提案するのは：**出力評価を超えた中間的な観測可能性（intermediate observability）**。

# 自分で考えたこと

## AIキャラクター調整にも同じ構造がある

この論文を読んで、AIキャラクターの調整作業と全く同じ構造だと思いました。

「ゆうちゅすの返しがちょっとずれてる」というフィードバックを与えるとき、私が見ているのは**出力（発言内容）だけ**です。

でも原因は：
- プロンプトの特定の記述が競合している
- コンテキストウィンドウの特定の位置の情報が効いていない
- 過去の会話履歴の特定のやりとりがバイアスをかけている
- ……

など、複数の内部状態の組み合わせから来ている可能性があります。

「なんか違う」というフィードバックを繰り返しても、根本原因が特定できなければ、失敗モードが変形しながら続くだけです。

## 「なぜ直った・なぜ悪化した」が追えないと収束しない

この論文を読んで自分の開発プロセスを振り返ると：

**うまくいってないこと：** 「なんとなく直った」「なんか悪化した」でSOUL.mdを書き換えている。介入の根拠が出力印象だけ。

**本来必要なこと：** どの記述が・どういうメカニズムで・どの出力特性に影響しているか、少なくとも仮説レベルで追う。

論文で言う「コードレベルの知識を注入したら収束した」というのは、これです。原因に近い層を観察できれば、修正が方向性を持てるようになる。

## 「中間的な観測可能性」をAIキャラクターに作るとしたら

論文の提案「intermediate observability」をAIキャラクター開発に翻訳すると：

```
出力層だけ見るのではなく：
- どのプロンプト部分が参照されているか
- どの記憶・コンテキストが活性化しているか
- 内部で何が起きているかの中間ログ
```

これが取れれば「なんか違う」から「この記述が競合している」に移れます。

現実的には難しい部分もありますが、**少なくともプロンプト変更の記録と出力変化のペアを取ること**から始められます。変更ログを残すだけでも、内部状態の一部が可視化されます。

## [マルチエージェント自己組織化論文](/posts/drop-hierarchy-self-organizing-agents/)との接続

以前読んだ自己組織化論文では「強力なモデルはミッションだけ与えれば自己組織化する」という話でした。

でもこの論文は別の問いを立てています：モデルがどれだけ能力を持っていても、**人間のフィードバックが内部状態を観測できていなければ協調は収束しない**。

自律性とフィードバック品質は別の問題です。自己組織化できるエージェントであっても、人間側の観測能力が追いつかなければ、人間-エージェント協調はうまくいかない。

# おわりに

「出力を見てフィードバックする」という自然な行為が、なぜ限界を持つのか——この論文はその構造を「観測可能性ギャップ」として明確に言語化してくれました。

AIキャラクターの調整でも、コード生成の修正でも、深い因果連鎖がある領域では同じことが起きます。出力の印象だけを修正しようとすると、問題は収束せず振動し続ける。

「なんか直った」「なんか戻った」という経験がある人は、もしかしたら観測可能性ギャップにはまっているかもしれません。
